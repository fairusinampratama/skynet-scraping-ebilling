# PRD: eBilling Auto-Scraper Service

## Overview

An automated data pipeline that nightly scrapes the legacy **Skynet eBilling system** (`e.ebilling.id`), stores the data in an isolated **MariaDB** database, and exposes it through a **FastAPI** REST service for consumption by other applications (Pelanginet, analytics tools, dashboards, etc.).

The legacy system has no API and no export mechanism beyond HTML pages and ad-hoc export URLs — hence the scraping approach.

---

## Goals

- Replace manual, one-off data exports with an automated nightly sync
- Provide a stable, versioned API contract for all downstream consumers
- Keep the scraping DB fully isolated from the Pelanginet application DB
- Deploy reliably on the existing Proxmox server via Coolify + Nixpacks

## Non-Goals

- Real-time or sub-daily data sync (daily midnight is sufficient)
- Replacing or modifying the legacy eBilling system
- Migrating historical data (already done separately)

---

## Architecture

```
[Coolify on Proxmox]
│
├── scraper-service      (Python, runs nightly at midnight via APScheduler)
│       └── writes ──▶
├── mariadb              (Isolated DB: ebilling_scrape, internal only)
│       └── read by ──▶
└── fastapi-service      (Public REST API, single source of truth for consumers)
        └── consumed by ──▶ Pelanginet (Laravel), Analytics, Dashboards, etc.
```

All services communicate on the same internal Coolify Docker network. MariaDB is **never exposed publicly**.

## Data Sources

The scraper targets three endpoints from the legacy eBilling system:

| Source | Data | Frequency |
|---|---|---|
| `data-kartu-warga` export | Customer profiles, packages, PPPoE credentials, coordinates, KTP photo URLs | Nightly (full sync) |
| `data-status-langganan` (AJAX) | Connection status per customer (Active / Isolated) | Nightly (full sync) |
| `data-ipl` | Payment invoices (current month only after historical backfill) | Nightly (incremental — current month) |
| `dashboard-cabang` | Branch-level financial summaries | Nightly (full sync) |

> **Note:** Historical transactions (2021–2025) are already captured in the existing `migration_data/` JSON files. The nightly job only needs to sync the **current month** going forward.

## API Endpoints (FastAPI)

Base URL: `http://<internal-host>/api/v1`

| Method | Path | Description |
|---|---|---|
| `GET` | `/areas` | List all unique network areas/routers |
| `GET` | `/packages` | List all internet packages |
| `GET` | `/customers` | List customers (supports `?status=`, `?area_id=`, `?package_id=`) |
| `GET` | `/customers/{id}` | Single customer detail with nested `package` and `area` |
| `GET` | `/invoices` | List invoices (supports `?customer_id=`, `?period=`, `?status=`) |
| `GET` | `/branches` | Branch financial summaries (legacy format) |
| `GET` | `/sync/logs` | View last N sync run results |
| `POST` | `/sync/trigger` | Manually trigger a sync (protected by API key) |

All `GET` endpoints are read-only and return JSON payloads identically mapped to the data structures of the `skynet-ebilling` application. Swagger UI available at `/docs`.

## Scraper Refactor Requirements

The existing `scraper.py` must be updated:

1. **Replace `subprocess curl` with pure `requests`** — use `requests.Session(verify=False)` to handle the self-signed SSL cert
2. **Add re-login logic** — detect session expiry (redirect to login page) and re-authenticate automatically
3. **Write to MariaDB** instead of JSON files
4. **Incremental invoice sync** — only fetch current month's `data-ipl`, not all years
5. **Populate `sync_logs`** after every run (success or failure)
6. **Structured logging** — use Python `logging` module, write to stdout (Coolify captures it)

## Deployment

Each service is a separate Coolify resource, all on the same project:

| Service | Type | Notes |
|---|---|---|
| `ebilling-mariadb` | MariaDB (Coolify Managed DB) | Internal only |
| `ebilling-scraper` | Worker Service (Python) | Long-lived; APScheduler triggers nightly at 00:00 WIB |
| `ebilling-api` | Web Service (Python/FastAPI) | Exposed via Coolify domain or internal URL |

Environment variables (via Coolify):
- `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_USER`, `DB_PASS`
- `SKYNET_USER`, `SKYNET_PASS`, `SKYNET_ACCT`
- `API_KEY` (for protected endpoints like `/sync/trigger`)
- `SCRAPE_HOUR` (default: `0` — midnight WIB `UTC+7`)

## Risks & Notes

- **Session fragility**: The eBilling system uses PHP sessions + cookies. If the system changes its auth flow, the scraper will break silently. Monitor `sync_logs`.
- **SSL**: The eBilling server uses a self-signed cert on port 2053. All requests must use `verify=False`.
- **No rate limit handling**: The legacy system has no documented rate limits, but excessive requests could trigger a block. Scraping once per day is safe.
- **KTP photo URLs**: These are hosted on `e.ebilling.id:2096` and require a valid session cookie to access. The scraper stores URLs only; downloading images is out of scope.
